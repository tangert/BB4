A linkedlist is a linear collection of elements, called nodes, that point to the next node through a pointer.



For the purpose of this tutorial, we will look at singly linkedlists. Singly linkedlists have a reference to the node after them, not before. A linkedlist where each node has a reference before and after it are called doubly linkedlists. In special cases, some doubly linkedlists have nodes where the last node has a reference to the first node. These are called circular linkedlists.

Popular methods:

  • Add to head: adds a node at the front of the list
  • Add to tail: adds a node at the end of the list
  • Add at index: adds a node at a specified index
  • Remove from head: removes the first node in the list
  • Remove from tail: removes the last node in the list
  • Remove at index: removes a node at a specified index
  • Get: returns the data in a specific node
  • Is empty: returns if the list is empty or not
  • Size: returns the size of the list


In real life, singly linkedlists can be represented easily by something like a conga line. Each person in the conga has their hands extended to the shoulders of the next person. Each person thus has a reference to the next person in front of them.

More abstractly, a linked list can be represented by a treasure map where each clue is a node. Each clue hints to the next clue and is thus essentially a pointer to the next clue.
A stack is a collection of elements that resembles a stack of something. The last element that is added becomes the first element to be removed. This is known as Last In, First Out, or LIFO.



For this tutorial, we will implement a stack using a linkedlist or array depending on the selected language.

Popular methods:

  • Push: pushes an element on top of the stack
  • Pop: removes the last element on the top of the stack
  • Peek: returns the value of the last element on top of the stack
  • Size: returns the number of elements in the stack


A real life example of a stack can be represented by a stack of cards. Each time a new card is added, it gets added to the top of the card stack. Each time a card is removed, the card on top is removed. Pretty simple.
A queue is a collection of elements that represents a queue or line of elements. The first element that is added to the queue becomes the first element to be removed. This is known as First In, First Out, or FIFO. In colloquial terms, this is also known as "First come, first serve."



For this tutorial, we will implement a stack using a linkedlist or array depending on the selected language.

Popular methods:

  • Enqueue: adds an element to the front of the queue
  • Dequeue: removes the first element in the queue
  • Peek: returns the value of the first element in the queue
  • Size: returns the number of elements in the stack


A real life example of a queue is, well, a music queue on something like Spotify. Each song that is added to the queue is the first one to be listened to when the queue is played. Another example would be standing in line at the grocery store. Customers enter a queue– a line to purchase their grocieries– and move through the queue until they've purchased everything.
A tree is a collection of elements presented in the form of a hierarchical tree. A tree starts with a root node and all other nodes have references to their "parent" and "child" nodes.



One of the most common tree structures is the Binary search tree, so we will focus on that in this implementation.

Trees are pretty tricky to come up with a "general" implementation for because every node in a tree can itself be considered a tree, and there are many different types of trees. As in this tutorial, there is the Binary search tree, the Red-Black tree, Cartesian tree...the list goes on and on. Below are some basic definitions to clear up any misunderstandings. Other tree structures include heaps and tries.

Basic definitions:

  • Binary tree: a tree where each parent node has at most two children nodes. This is the base for many commonly used tree structures
  • Binary search tree: a binary tree that is specifically geared towards serching. A node's left child contains only nodes with values less than the parent node, and the right child only contains nodes with values greater than or equal to the parent.
  • Red-Black tree: a self-balancing binary search tree where each node is either red or black, represented by an extra bit of information. These color bits are used to ensure the tree remains approximately balanced during insertions and deletions.

Popular methods:

  • Insert: adds a node into the tree
  • Remove: removes a specified node from the tree
  • Find min: returns the minimum value in the tree
  • Find max: returns the maximum value in the tree
  • Find: finds a specified node in the tree
  • Make empty: empties out the tree


To give a real life example, let's look at a family tree. If we start with a grandfather, we can traverse down the tree and see that grandpa has two children: a son, and a daughter. That son has his two of his own children, and that daughter also has two of her own children. The reason trees are composed of "subtrees" is because, from grandpa's perspective, he has children and grandchildren. However, from his children's perspective, he isn't grandpa; he's just dad.. Therefore the entire family tree is composed of various parent-child relationships that are independent of each other. Grandpa is just "dad" to dad, and dad is "dad" to his children as well.
A graph is a collection of elements called vertexes connected together by edges. Another name for graphs in everyday terms is a network.



Graphs can (pretty much) either be undirected or directed, and weighted or unweighted. Therefore there are 4 common combinations of graphs:

  1. Undirected unweighted graph
  2. Undirected weighted graph
  3. Directed unweighted graph
  4. Directed weighted graph
In our example, we are going to look at an undirected, unweighted graph because it's the simplest representation of a graph.

Basic definitions:

  • Undirected graph: graphs edges don't have a specific direction. Each vertex can travel to another vertex and back. They are bidirectional. Think a game of catch between three people, where each person is a vertex and the edge is the ball path. Each person can throw to the other in any combination.
  • Directed graph: graphs where edges have a specific direction. Each node can only travel one way to another node. They are unidirectional. Take the same game of catch between three people, except person 1 can only throw to person 2, person 2 to person 3, and person 3 only back to person 1.
  • Weighted graph: graphs where edges have specific weights/costs associated with them. In practice, let's say that we have two vertices: New York, and San Francisco. If they are in a weighted graph connected by an edge, that edge may represent the length between the two cities, so it's weight would be 3000 if our units are miles. It may be 6 if our unit is hours flown by airplane.
  • Unweighted graph: graphs where edges don't have specific weights/costs associated with them. Taking the New York to San Francisco example, the edge would simply represent the fact that there is some sort of route between the two cities without any data on what that route is.


Popular methods:

  • Add vertex: adds a vertex to the graph
  • Remove vertex: removes a vertex from the graph
  • Add edge: adds an edge to the graph
  • Remove edge: removes an edge from the graph
  • Adjacent: checks if two vertices are adjacent


A real life example of a graph, as said, can be any sort of network. A few people playing a game, a social network (like your facebook friends), a map with various routes, etc. Graphs are inherently pretty intuitive once you understand their constiuent components.
